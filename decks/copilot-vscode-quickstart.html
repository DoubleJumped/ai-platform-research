02 / 12

when to change tier

Default Sol medium. Escalate on signals.

^ Sol high when

  • A cheaper model failed the same error twice
  • The task spans multiple files or subsystems
  • Requirements are ambiguous
  • The run will exceed ~10 minutes

v Luna high for

  • Repo search and navigation
  • Boilerplate and test scaffolding
  • Well-defined single-file edits
  • Summaries, changelogs, comments

Drop back to medium when the hard part is done. Higher effort bills as more reasoning tokens — there is no multiplier, but there is no refund either.

03 / 12

part two // the method

One task per context window

Model recall degrades as the window fills, and every request re-bills the whole thread. So: plan once, write the tasks to a file, implement one task per fresh window.

This is the documented best practice, not a house invention — Anthropic's guidance is literally “write the spec to a file, then start a fresh session to execute it.”

step 2 // persist

Write the tasks to plan.md

Copilot keeps the plan in session memory — it's deleted when the conversation ends. Tell it to write a checklist to the repo instead.

  • The register is the state that survives between windows
  • Tasks, subtasks, and a notes section for learnings
plan.md
> write this plan to plan.md as a task checklist

## rate limiting
- [x] 1. middleware skeleton + config
- [ ] 2. redis counter, sliding window
  - [ ] 2a. key schema + ttl
  - [ ] 2b. 429 + Retry-After header
- [ ] 3. tests: 10/min pass, 11th = 429
## notes
- reuse RedisClient from src/cache.ts

The loop: fresh window, one task

1
/clear — start a fresh window
every task
2
Load context: #file:plan.md + exemplar files — "implement task 2 only"
Sol medium
3
Implement + verify: tests, lint, run the changed behavior
Sol medium
4
Update the register: check off the task, append learnings to notes
then repeat

Each window starts clean and reads its state from plan.md. No window inherits another window's junk.

Point at an example, not at the repo

The best convention document is a file that already follows the convention. Pin it. Don't say “investigate the codebase” — unscoped exploration fills the window with junk.

08 / 12

context // know when to leave

At ~40% full: synthesize, then restart

The chat input shows a context gauge — hover for exact tokens (e.g. 46K/128K) and what's using them. Quality decays long before the window is full. Don't ride it down: write what you've learned into plan.md notes, /clear, continue fresh.

~40%

House rule: finish the thought, write learnings to the register, restart. Vendors publish no number; degradation is measurable well before the limit.

/compact

The fallback when you must stay in a thread: summarizes it in place (add focus: /compact focus on the schema).

2x

Billing past 272K input on GPT-5.6: the whole request re-prices. Long threads get dumber and more expensive at the same time.

10 / 12

finish // two checks

Two checks before you trust a run

Set the permission level on purpose

  • Autopilot is now the default (June 2026): self-approves actions and runs to completion
  • Per session: Default / Bypass Approvals / Autopilot — pick deliberately, especially near configs, deletes, or migrations

A human reviews every diff

  • Inspect the diff against the task's acceptance criteria; reject unrelated changes
  • An agent review can help — give it only the diff + criteria, flag correctness gaps only — but it never replaces you

Recovery, in order: keep/undo individual edits → restore checkpoint → /fork to try another path → /clear when the context is polluted.

caveats

  • Edit mode is deprecated: hidden since Jan 2026, removal scheduled — use Agent for multi-file edits
  • The ~40% rule is house guidance; vendors publish no threshold
  • Writing the plan to a repo file is an instruction you give, not documented VS Code behavior
  • Real slash commands: /plan /compact /clear /fork /fix /tests /init /new — there is no /resume
  • Sonnet 5 pricing is a promo ending 2026-08-31
  • Features and prices move monthly; recheck before circulating
